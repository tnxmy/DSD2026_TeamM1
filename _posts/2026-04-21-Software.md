## 0. Background

### 0.1 Purpose of This Document

This document defines the Software Requirements Specification for **Team M1**, responsible for the **Patient Mobile Application** within the Monitor Layer of the Limb Motion Recognition and Assistant system.

This document serves three purposes:
- **Part I (Sections 0–5)**: Describes what the app does from the patient's perspective.
- **Part II (Section 6)**: Describes how internal modules interact to fulfil the external requirements, without specifying concrete interface implementations.
- **Part III (Section 7)**: Defines quality attributes applicable to the entire system.

Interface design and system architecture are documented separately and are not part of this SRS.

## Part II: Internal Requirements

### 6. Internal Use Cases

This section describes the interactions between internal modules of the M1 application and external system modules (V2 Backend, Sensor Layer) that are necessary to fulfil the external requirements defined in Part I. These use cases are for internal team coordination and do not specify concrete interface implementations such as REST endpoints, WebSocket paths, or JSON schemas. Interface design will be documented separately.

#### 6.1 Internal Actor Definitions

| Actor | Role | Description |
| :------------: | :------------: | :------------: |
| M1 App | Internal System | The patient mobile application itself, comprising UI layer, business logic layer, and data access layer. |
| V2 Backend | External System | The server-side system responsible for authentication, data storage, exercise plan management, and AI result delivery. |
| Sensor Layer | External System | The hardware and firmware layer (S1, S2) that provides raw motion data via Bluetooth Low Energy. |
| Push Service | External System | The third-party notification delivery service used to push messages to the patient device. |

| IUC-M1-05 | Stream Real-Time Session Data |
| :------------: | :------------: |
| Actor | M1 App, V2 Backend, Sensor Layer |

## Part III: Non-Functional Requirements

### 7. Non-Functional Requirements

#### 7.1 Performance and Latency

| ID | Requirement |
| :------------: | :------------: |
| NFR-M1-01 | Real-time visualisation must update within **200ms** of each incoming data frame to maintain perceived immediacy. |
| NFR-M1-02 | Posture alerts must be displayed within **100ms** of receiving an alert payload. |
| NFR-M1-03 | App launch to login screen must take less than **3 seconds** on a mid-range device. |
| NFR-M1-04 | Server requests must have a client-side timeout of **10 seconds** with a clear error message on expiry. |
| NFR-M1-05 | The 3D rendering pipeline must maintain at least **30 frames per second** on the target device class. |

#### 7.2 Usability

| ID | Requirement |
| :------------: | :------------: |
| NFR-M1-06 | The app must be usable by patients with **no prior technical knowledge**. No technical terminology shall be exposed in the patient-facing UI. |
| NFR-M1-07 | All critical actions (start session, end session, connect sensor) must be reachable within **2 taps** from the dashboard. |
| NFR-M1-08 | Posture correction messages must be written in plain language: e.g., *"Bend your knee more"* rather than *"Deviation: -24.8°"*. |
| NFR-M1-09 | Font sizes and contrast ratios must meet **WCAG AA** accessibility standards. |
| NFR-M1-10 | Error messages must always offer a next step. **No dead-end error states** permitted. |

#### 7.3 Security

| ID | Requirement |
| :------------: | :------------: |
| NFR-M1-11 | All network communication must use **encrypted transport**. |
| NFR-M1-12 | Session tokens must be stored using platform secure storage (**iOS Keychain / Android Keystore**). |
| NFR-M1-13 | No biometric or health data may be **logged to device console or crash logs**. |
| NFR-M1-14 | The app must comply with **GDPR** requirements for health data. Patients must consent before data collection begins. |
| NFR-M1-15 | Session tokens must be **invalidated on logout**. |

#### 7.4 Reliability

| ID | Requirement |
| :------------: | :------------: |
| NFR-M1-16 | Data stream disconnection during an active session must trigger **automatic reconnection with exponential backoff**. Patient notified if reconnection fails after 30 seconds. |
| NFR-M1-17 | BLE disconnection must not cause **data loss** for frames already received and processed. |
| NFR-M1-18 | The app must handle **intermittent network conditions gracefully** without crashing. |

#### 7.5 Maintainability

| ID | Requirement |
| :------------: | :------------: |
| NFR-M1-19 | The app must be built with **React Native (Expo) + TypeScript** following M1 architectural conventions. |
| NFR-M1-20 | All components must be tested by the **cross-testing model** defined in the team methodology. |
| NFR-M1-21 | All commit messages, PR descriptions, and code comments must be written in **English**. |

### 8. Assumptions and Open Items

#### 8.1 Assumptions

| ID | Assumption |
| :------------: | :------------: |
| A-M1-05 | The IMU sensors communicate via **Bluetooth Low Energy**. M1 does not handle WiFi-based sensor protocols. |
| A-M1-06 | Exercise plans are created exclusively in **M2 by doctors** and delivered to M1 via V2. M1 has no plan-creation functionality. |
| A-M1-07 | Push notification delivery is handled via an external service configured by V2. M1 only handles registration and display. |
| A-M1-08 | Target devices run **iOS 14+** or **Android 9+**. |
| A-M1-09 | IF2 interface contract will be frozen by the end of **Sprint 2**. M1 will use mock data following the agreed schema until the contract is confirmed. |

#### 8.2 Open Items

| Item | Status | Resolution Path |
| :------------: | :------------: | :------------: |
| Exact real-time data endpoint specification | Open | Joint M1–V2 meeting before end of Sprint 2 |
| Joint naming convention in motion data payload | Open | Align with V1 model output schema |
| Alert severity thresholds (clinical validation) | Open | Input required from clinical team or reference values |
| 3D rendering library selection | Internal | M1 internal decision — resolve by Sprint 2 kick-off |
| Push notification service provider | V2 Decision | V2 confirms provider; M1 integrates SDK accordingly |
