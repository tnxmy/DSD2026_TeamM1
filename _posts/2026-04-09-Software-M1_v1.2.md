## 0. Background

### 0.1 Purpose of This Document

This document defines the Software Requirements Specification for **Team M1**, responsible for the **Patient Mobile Application** within the Monitor Layer of the Limb Motion Recognition and Assistant system.

This document serves three purposes:

<div style="margin-left: 2rem;">
<ul style="list-style: disc; margin: 0.5rem 0;">
<li><strong>Part I (Sections 0–4)</strong>: Describes what the app does from the patient's perspective.</li>
<li><strong>Part II (Section 5)</strong>: Describes how internal modules interact to fulfil the external requirements, without specifying concrete interface implementations.</li>
<li><strong>Part III (Section 6)</strong>: Defines quality attributes applicable to the entire system.</li>
</ul>
</div>

Interface design and system architecture are documented separately and are not part of this SRS.

## 5. Internal Use Cases

### 5.1 Internal Actor Definitions

<div class="actor-grid">
  <div class="actor-card">
    <div class="actor-icon cyan-icon">⚙️</div>
    <span class="actor-tag" style="color:var(--cyan);">Internal System</span>
    <h4>S1 (Sensor Link)</h4>
    <p>Sensor connection module inside M1. Responsible for discovering and establishing a connection with the physical sensor and receiving raw motion data broadcasts from the Sensor Layer. S1 passes raw data to other components within M1 for further processing; it does not communicate directly with external systems over the network.</p>
  </div>
  <div class="actor-card">
    <div class="actor-icon cyan-icon">⚙️</div>
    <span class="actor-tag" style="color:var(--cyan);">Internal System</span>
    <h4>S2 (Sensor Data Processing)</h4>
    <p>Sensor data processing module inside M1. Receives raw motion data from S1, performs validation and filtering, computes target joint angles, detects motion deviations, and assembles processed results into a structured representation of sensor data. S2 manages the lifecycle state of a rehabilitation session.</p>
  </div>
  <div class="actor-card">
    <div class="actor-icon green-icon">🧑‍⚕️</div>
    <span class="actor-tag" style="color:var(--green);">Internal System</span>
    <h4>M1 (App Subject)</h4>
    <p>The patient mobile application as a whole. M1 encapsulates S1 and S2 as internal constituents and presents itself as a single entity to the outside world. M1 coordinates UI, sensor communication, data processing, and all data exchanges with external systems.</p>
  </div>
  <div class="actor-card">
    <div class="actor-icon blue-icon">🔵</div>
    <span class="actor-tag" style="color:var(--blue);">External System</span>
    <h4>V2 (Backend)</h4>
    <p>The central backend server that provides persistent storage for user accounts, rehabilitation sessions, measurement data, and AI-generated recommendations. It exposes a service interface for data exchange with M1.</p>
  </div>
  <div class="actor-card">
    <div class="actor-icon orange-icon">📡</div>
    <span class="actor-tag" style="color:var(--orange);">External System</span>
    <h4>Sensor Layer</h4>
    <p>The physical wearable sensor hardware that communicates wirelessly with M1, providing raw motion data and receiving connection commands.</p>
  </div>
  <div class="actor-card">
    <div class="actor-icon blue-icon">🔔</div>
    <span class="actor-tag" style="color:var(--blue);">External System</span>
    <h4>Push Service</h4>
    <p>A third-party notification delivery service that receives dispatch instructions from V2 and delivers notification messages to the patient device.</p>
  </div>
</div>
