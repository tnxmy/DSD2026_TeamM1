## 0. Background

### 0.1 Purpose of This Document

This document defines the Software Requirements Specification for **Team M1**, responsible for the **Patient Mobile Application** within the Monitor Layer of the Limb Motion Recognition and Assistant system.

The purpose of this document is to:

- Define what M1 is responsible for building and what it is not.
- Specify the use cases for the Patient actor in the mobile context.
- Define the technical requirements for consuming the IF2 interface from V2.
- Align M1's functional and non-functional requirements with the rest of the system.

This document is intended to be read together with the system-level SRS. M1 does not redefine system-wide use cases; it specialises and expands them for the mobile patient context.

## 1. Actors

| Actor | Type | Description |
| :------------: | :------------: | :------------: |
| Patient | Primary | A rehabilitation patient who wears IMU sensor hardware during exercise sessions and uses the mobile application as their primary interface to the system. |
| System | Secondary | V2 Backend API & Storage. Provides all data to M1 via IF2. Authenticates the patient and issues JWT tokens. |

## 3. Interface Requirements — IF2

IF2 is the formal interface contract between the **Server Layer (V2)** and the **Monitor Layer (M1, M2)**. M1 co-owns this contract alongside V2.

### 3.1 IF2 Overview

| Property | Value |
| :------------: | :------------: |
| Contract Owners | V2 + M1 |
| Real-Time Protocol | WebSocket (WSS) |
| Historical / Management | HTTPS REST |
| Authentication | JWT Bearer Token |
| Data Format | JSON |
| Contract Freeze | End of Sprint 2 |

### 3.2 Authentication Requirements

| Requirement | Description |
| :------------: | :------------: |
| Token Issuance | V2 issues JWT on login. Included as `Authorization: Bearer &lt;token&gt;` in REST and WebSocket handshake. |
| Token Expiry | Configurable expiry. M1 handles refresh via `POST /auth/refresh`. |
| Token Storage | iOS Keychain / Android Keystore. Never plain-text. |

### 3.3 REST Endpoints Required by M1

| Method | Endpoint | Purpose |
| :------------: | :------------: | :------------: |
| POST | /auth/register | Patient registration |
| POST | /auth/login | Patient login — returns JWT |
| POST | /auth/refresh | Token refresh |
| POST | /auth/reset-password | Password reset request |
| POST | /sessions/start | Start session |
| POST | /sessions/{id}/end | End session |
| GET | /sessions/{id}/summary | Session summary |
| GET | /sessions/history | Historical sessions (last 15 days) |
| DELETE | /sessions/{id} | Delete session record |
| GET | /plans/active | Active rehabilitation plan |
| GET | /patients/me/achievements | Achievement data |
| POST | /devices/connection | Report BLE sensor state |
| POST | /notifications/register | Register push token |

### 3.4 WebSocket Requirements

| Property | Value |
| :------------: | :------------: |
| Endpoint Pattern | `wss://{server}/sessions/{session_id}/stream` |
| Connection Trigger | After `POST /sessions/start` response |
| Message Direction | V2 → M1 (server-push). M1 sends heartbeat pings only. |
| Target Frequency | One payload per frame (~100ms) |
| Reconnection Strategy | Exponential backoff, max 3 retries |

### 3.5 IF2 Payload Schema (Real-Time Stream)

```json
{
  "session_id": "uuid-string",
  "timestamp": "2026-04-06T14:00:00.000Z",
  "recognised_angles": {
    "knee_flex": 87.4,
    "hip_flex": 45.1,
    "ankle_dorsi": 12.3
  },
  "target_angles": {
    "knee_flex": 90.0,
    "hip_flex": 45.0,
    "ankle_dorsi": 10.0
  },
  "deviation_score": 0.93,
  "alert": {
    "joint": "knee_flex",
    "type": "under_target",
    "current_angle": 65.2,
    "target_angle": 90.0,
    "message": "Bend your knee further",
    "severity": "warning"
  },
  "confidence": 0.98,
  "model_version": "v2.1"
}
```

|     Types     | Session reminder, Plan updated, Doctor message, Milestone |
|  User Action  |    Tap to navigate to relevant screen, Dismiss to clear   |
| Local Storage |           Notification history retained 30 days           |

### 6.2 Data Lifecycle Management

|   Stage   |                    Action                    |        Trigger        |
| :-------: | :------------------------------------------: | :-------------------: |
|  Creation |  Session data stored locally + synced to V2  |      Session end      |
| Retention |   Full data (including 3D) kept for 15 days  |       Time-based      |
|  Archive  |     Summary-only retained, 3D data purged    |     After 15 days     |
|  Deletion |      User-initiated delete synced to V2      |      User action      |
|  Cleanup  | Auto-remove local records older than 15 days | Daily background task |
